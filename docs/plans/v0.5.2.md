# v0.5.2 — Outcomes, Memory, Multi-Agent, Dreaming

> Scope: Four Anthropic MA capabilities that close the remaining feature gaps.
> v0.5.0 aligned API shapes. v0.5.2 adds the missing runtime features.
> Significant infrastructure already exists — this release wires it up
> to match the Anthropic API surface and fills the gaps.

## What Already Exists

| Feature | Existing Code | What's Missing |
|---------|--------------|----------------|
| **Outcomes** | `sessions/grader.ts` (121 lines) — direct Anthropic API grader call. `outcome_criteria_json` column on sessions. | `user.define_outcome` event handling, evaluation loop in driver, `span.outcome_evaluation_*` events, `outcome_evaluations[]` on session response, rubric file support |
| **Memory** | `db/memory.ts` (212 lines) — full CRUD for stores + memories with SHA256 OCC. `handlers/memory.ts` (180 lines) — all 9 HTTP endpoints working. Schema: `memory_stores` + `memories` tables. | Container mount at `/mnt/memory/`, memory versions table + audit trail, `memory_store` resource type on sessions, redaction, version listing |
| **Multi-Agent** | `sessions/threads.ts` (176 lines) — `handleSpawnAgent()` spawns child sessions with depth limits, cost rollup, event emission. `handlers/threads.ts` — list endpoint. `callable_agents` + `threads_enabled` on agents. | `multiagent` config field on agents (replaces `threads_enabled`+`callable_agents`), thread-level event streams, primary thread aggregation events, thread archive endpoint |
| **Dreaming** | Nothing | Scheduled job, session review, memory curation |

## Non-goals

- Memory versioning redaction (compliance feature, defer to v0.6)
- Recursive multi-agent (depth > 1 already works internally, but Anthropic limits to 1)
- Dreaming configuration UI (API + cron only)
- Files API (`/v1/files`) — we have container file sync instead

## PR 1 — Outcomes

### What it does

Send `user.define_outcome` to a session. The agent works toward the rubric. After each iteration, a grader evaluates output. If unsatisfied, feedback loops back. Events stream progress.

### Schema changes

```sql
ALTER TABLE sessions ADD COLUMN outcome_id TEXT;
-- outcome_criteria_json already exists
```

### Event types to add

| Event | When | Key fields |
|-------|------|------------|
| `user.define_outcome` | User sends it | `description, rubric, max_iterations, outcome_id` |
| `span.outcome_evaluation_start` | Grader begins | `outcome_id, iteration` |
| `span.outcome_evaluation_ongoing` | Grader heartbeat | `outcome_id` |
| `span.outcome_evaluation_end` | Grader finishes | `outcome_id, result, explanation, iteration, usage` |

Results: `satisfied`, `needs_revision`, `max_iterations_reached`, `failed`, `interrupted`

### Files to modify

- `packages/agent-sdk/src/handlers/events.ts` — Accept `user.define_outcome` event type. Parse rubric (inline text or file_id). Store rubric + description in `outcome_criteria_json`. Generate `outcome_id` (`outc_` prefix). Kick off first turn.
- `packages/agent-sdk/src/sessions/driver.ts` — After agent stop (`end_turn`), check if session has an active outcome. If yes: emit `span.outcome_evaluation_start`, call `runGraderEvaluation()` from `grader.ts`, emit `span.outcome_evaluation_end`. If `needs_revision` and iterations < max, send grader feedback as next turn input and continue. If `satisfied` or `max_iterations_reached`, transition to idle.
- `packages/agent-sdk/src/sessions/grader.ts` — Already exists. Extend to accept rubric as text (current) or file content. Add iteration tracking. Return structured `GraderResult` with explanation.
- `packages/agent-sdk/src/db/sessions.ts` — Add `outcome_id` to hydration. Add `outcome_evaluations` array to session response (query evaluation events from DB).
- `packages/agent-sdk/src/types.ts` — Add `OutcomeEvaluation` interface. Add `outcome_evaluations: OutcomeEvaluation[]` to Session.
- `packages/agent-sdk/src/util/ids.ts` — Add `outc_` prefix.

### Implementation flow

```
user.define_outcome received
  → store rubric in outcome_criteria_json
  → generate outcome_id (outc_...)
  → echo event back with outcome_id + processed_at
  → description becomes the first turn's user message
  → agent works (normal turn loop)
  → agent stops (end_turn)
  → driver checks: has active outcome?
    → yes: emit span.outcome_evaluation_start
    → call grader with rubric + agent's last output
    → emit span.outcome_evaluation_end with result
    → if needs_revision && iteration < max:
      → inject grader feedback as next user message
      → continue turn loop (iteration++)
    → if satisfied || max_iterations_reached || failed:
      → session → idle
```

### Tests

- Create session, send `user.define_outcome`, verify events stream
- Grader returns `satisfied` on first try → session goes idle
- Grader returns `needs_revision` → agent iterates → verify iteration count
- `max_iterations` respected
- `user.interrupt` during evaluation → result = `interrupted`
- `outcome_evaluations[]` populated on GET /sessions/:id

## PR 2 — Memory (container mount + versions)

### What it does

Memory stores mount into containers at `/mnt/memory/<store-name>/`. Agent reads/writes with file tools. Changes sync back to the DB. Every mutation creates an immutable version.

### Schema changes

```sql
CREATE TABLE IF NOT EXISTS memory_versions (
  id TEXT PRIMARY KEY,
  store_id TEXT NOT NULL,
  memory_id TEXT NOT NULL,
  operation TEXT NOT NULL,  -- 'create' | 'update' | 'delete'
  path TEXT NOT NULL,
  content TEXT,
  content_sha256 TEXT,
  session_id TEXT,
  created_at INTEGER NOT NULL,
  FOREIGN KEY (store_id) REFERENCES memory_stores(id)
);
CREATE INDEX idx_memver_store ON memory_versions(store_id, created_at);
CREATE INDEX idx_memver_memory ON memory_versions(memory_id, created_at);
```

### Session resource: `memory_store` type

Accept in `resources[]` on session create:
```json
{
  "type": "memory_store",
  "memory_store_id": "memstore_...",
  "access": "read_write",
  "instructions": "Check before starting."
}
```

Max 8 memory stores per session.

### Container mount

On session start, for each memory_store resource:
1. Read all memories from the store
2. Write them to `/mnt/memory/<store-name>/<path>` in the container
3. Add mount info to the system prompt injection
4. After each turn, diff `/mnt/memory/` against DB state, sync changes back
5. Each sync creates a `memory_version` row

### Files to modify

- `packages/agent-sdk/src/handlers/sessions.ts` — Accept `memory_store` resource type in CreateSchema. Validate store exists and access mode.
- `packages/agent-sdk/src/containers/lifecycle.ts` — On first turn, mount memory stores into container. After turn completion, sync changes back.
- `packages/agent-sdk/src/db/memory.ts` — Add `createMemoryVersion()`, `listMemoryVersions()`, `getMemoryVersion()`. Update `createOrUpsertMemory()` and `deleteMemory()` to emit versions.
- `packages/agent-sdk/src/handlers/memory.ts` — Add `handleListMemoryVersions()`, `handleGetMemoryVersion()`. Add `handleArchiveMemoryStore()`.
- `packages/agent-sdk/src/db/migrations.ts` — `memory_versions` table.
- `packages/agent-sdk/src/util/ids.ts` — Add `memver_` prefix.
- `packages/agent-sdk/src/types.ts` — Add `MemoryVersion` interface. Add `memory_store` to `SessionResource` type.

### Tests

- Create memory store, attach to session, verify files appear in container
- Agent writes to `/mnt/memory/`, verify DB updated with new memory + version
- `read_only` access prevents writes (filesystem-level)
- Memory versions list shows create/update/delete history
- SHA256 OCC on update prevents clobber

## PR 3 — Multi-Agent (Anthropic API alignment)

### What it does

Coordinator agent delegates to specialist agents via threads. Each thread is a context-isolated event stream sharing the container filesystem.

### What changes from existing implementation

The existing `handleSpawnAgent()` creates a child *session* (separate container). Anthropic's model uses *threads* (same container, separate context). We need to align:

| Current | Anthropic | Change |
|---------|-----------|--------|
| `threads_enabled` + `callable_agents` on agent | `multiagent: { type: "coordinator", agents: [...] }` | Replace fields |
| Child gets own container | Threads share coordinator's container | Change container binding |
| `session.thread_started/completed` events | `session.thread_created`, `session.thread_status_*`, `agent.thread_message_*` | Add event types |
| No thread-level event stream | `/sessions/:id/threads/:tid/events` + `/stream` | Add endpoints |
| No thread archive | `POST /sessions/:id/threads/:tid/archive` | Add endpoint |

### Schema changes

```sql
CREATE TABLE IF NOT EXISTS session_threads (
  id TEXT PRIMARY KEY,
  session_id TEXT NOT NULL,
  agent_id TEXT NOT NULL,
  agent_version INTEGER NOT NULL,
  parent_thread_id TEXT,
  status TEXT NOT NULL DEFAULT 'idle',
  stop_reason TEXT,
  created_at INTEGER NOT NULL,
  archived_at INTEGER,
  FOREIGN KEY (session_id) REFERENCES sessions(id)
);
CREATE INDEX idx_threads_session ON session_threads(session_id);
```

### Agent config change

Replace `threads_enabled` + `callable_agents` with:
```json
{
  "multiagent": {
    "type": "coordinator",
    "agents": [
      { "type": "agent", "id": "agnt_...", "version": 1 },
      { "type": "self" }
    ]
  }
}
```

Max 20 agents in roster. Depth = 1 (coordinator only, no recursive).

### Files to modify

- `packages/agent-sdk/src/types.ts` — Add `multiagent` to Agent interface. Add `SessionThread` interface.
- `packages/agent-sdk/src/db/agents.ts` — Store `multiagent` in `agent_versions` (new column or reuse `callable_agents_json`).
- `packages/agent-sdk/src/handlers/agents.ts` — Accept `multiagent` in create/update schemas.
- `packages/agent-sdk/src/sessions/threads.ts` — Refactor `handleSpawnAgent()` to use same container as parent. Create thread row instead of child session. Run agent CLI in the shared container with thread-specific session ID.
- `packages/agent-sdk/src/handlers/threads.ts` — Add `handleGetThread()`, `handleArchiveThread()`, `handleListThreadEvents()`, `handleStreamThreadEvents()`.
- `packages/agent-sdk/src/sessions/driver.ts` — When coordinator calls the synthetic `spawn_agent` tool, route through the thread system. Inject thread events into the primary stream.
- `packages/agent-sdk/src/db/migrations.ts` — `session_threads` table. `multiagent_json` column on `agent_versions`.
- `packages/gateway-hono/src/index.ts` — Register thread endpoints.

### Event types

| Event | Primary thread? | Thread stream? |
|-------|----------------|----------------|
| `session.thread_created` | Yes | No |
| `session.thread_status_running` | Yes | Yes |
| `session.thread_status_idle` | Yes | Yes |
| `session.thread_status_terminated` | Yes | Yes |
| `agent.thread_message_received` | Yes | No |
| `agent.thread_message_sent` | Yes | No |
| `agent.*` (tool_use, message, etc.) | Condensed | Full detail |

### Tests

- Create coordinator with `multiagent`, verify config stored
- Coordinator spawns thread, verify thread events on primary stream
- Thread-level event stream shows full agent activity
- Thread archive works (only when idle)
- Tool permission requests cross-posted to primary thread
- Max 25 concurrent threads enforced
- Cost rollup from threads to parent session

## PR 4 — Dreaming (basic)

### What it does

A scheduled job reviews recent sessions and memory stores. Extracts patterns, curates memories. Operates between sessions, not during them.

### Implementation

A new CLI command: `gateway dream` that can be run via cron or scheduled job.

```bash
# Run nightly via cron
0 2 * * * gateway dream --memory-store memstore_... --lookback 24h
```

### How it works

1. Query sessions completed in the lookback window
2. For each session, extract key events (user messages, agent messages, errors, tool failures)
3. Send a curated prompt to Claude with the session summaries + current memory store contents
4. Claude analyzes patterns: recurring mistakes, workflow convergence, preferences
5. Claude outputs memory updates as structured JSON (create/update/delete operations)
6. Apply updates to the memory store (with version tracking)
7. Log what changed

### Files to create

- `packages/agent-sdk/src/dreaming/review.ts` — Core dreaming logic: session extraction, prompt construction, memory update application
- `packages/agent-sdk/src/dreaming/prompts.ts` — System prompt for the dreaming analysis

### Files to modify

- `packages/gateway/src/commands/dream.ts` — CLI command: `gateway dream --memory-store <id> --lookback <duration> --dry-run`
- `packages/gateway/src/index.ts` — Register command

### Configuration

```bash
# Via CLI
gateway dream --memory-store memstore_abc --lookback 24h --dry-run
gateway dream --memory-store memstore_abc --lookback 7d --auto-apply

# Via settings (for automated scheduling)
gateway settings set dreaming_enabled true
gateway settings set dreaming_lookback_hours 24
gateway settings set dreaming_memory_store memstore_abc
```

### Tests

- Session extraction produces correct summaries
- Dry-run mode shows proposed changes without applying
- Auto-apply creates memory versions with `operation: "dream"`
- Empty sessions produce no changes
